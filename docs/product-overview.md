# 제품 소개

Harness Docs는 고객-facing 문서를 운영하는 팀을 위한 문서 워크스페이스입니다.

이 제품은 단순히 문서를 저장하는 곳이 아니라, 아래 흐름을 한곳에서 연결하는 것을 목표로 합니다.

- 고객이 원래 어떤 문제를 겪고 있었는지 설명
- 왜 기존 방식이 불편했는지 공감 형성
- 우리 제품이 어떤 방식으로 문제를 풀어가는지 전달
- 시작하기, 설정, 운영, FAQ까지 실제 사용 가이드 제공

## 어떤 팀을 위한 제품인가

Harness Docs는 다음과 같은 팀을 위한 제품입니다.

- 제품을 소개하는 문서와 실제 사용 가이드를 함께 운영해야 하는 팀
- 고객 문제 설명, 기능 가치 전달, 온보딩 문서를 분리하지 않고 연결해 관리하고 싶은 팀
- GitHub를 문서 배포 채널로 쓰되, 문서 상태와 검토 이력은 앱에서 관리하고 싶은 팀

## 어떤 문제를 풀려고 하는가

많은 팀은 고객-facing 문서를 만들 때 아래 문제를 반복해서 겪습니다.

- 문제 설명 문서와 사용 가이드가 따로 놀아서 읽는 사람이 흐름을 놓침
- 문서가 오래되어도 어디를 먼저 고쳐야 하는지 판단하기 어려움
- 실제 배포되는 문서와 내부 검토 상태가 분리되어 운영 부담이 커짐
- 제품을 잘 아는 사람만 문서를 고칠 수 있고, 팀 공통 편집 흐름이 약함

Harness Docs는 이 문제를 문서를 "운영 대상"으로 다루는 방식으로 풀려고 합니다.

## 제품이 제공하는 핵심 가치

### 1. 문제 공감에서 사용 가이드까지 한 흐름으로 연결

문서를 읽는 사람은 먼저 "왜 이 제품이 필요한가"를 이해해야 하고, 그 다음 "어떻게 쓰는가"로 자연스럽게 넘어가야 합니다.

Harness Docs는 문제 설명, 도입 흐름, setup guide, 운영 문서를 같은 워크스페이스 안에서 연결해 관리합니다.

### 2. 고객-facing 문서 운영에 맞는 워크스페이스

각 워크스페이스는 하나의 GitHub docs 저장소와 연결됩니다. 팀은 이 안에서 문서를 작성하고 검토한 뒤, 배포 가능한 상태로 정리합니다.

즉 GitHub는 최종 배포 채널이고, 문서의 작업 맥락과 검토 이력은 앱이 관리합니다.

### 3. 팀이 같이 관리할 수 있는 문서 품질 흐름

문서는 한 번 쓰고 끝나는 자산이 아니라 계속 손봐야 하는 운영 대상입니다.

Harness Docs는 아래 같은 흐름을 제품 안에서 다룹니다.

- 어떤 문서가 최근 바뀌었는지
- 어떤 문서가 검토가 필요한지
- 어떤 문서가 업데이트가 필요한지
- 어떤 상태로 배포 준비가 되었는지

### 4. AI를 초안 도구로 사용

AI는 최종 결정을 대신하지 않습니다. 대신 아래 작업을 빠르게 시작하게 돕습니다.

- 고객 문제 설명 초안 작성
- 사용 가이드 구조 제안
- 문서 보강 포인트 정리
- 운영/FAQ 문안 출발점 생성

## 문서 예시

Harness Docs 안에서는 아래 같은 문서를 운영할 수 있습니다.

- 고객 문제와 제품 가치를 설명하는 문서
- 처음 쓰는 사용자를 위한 시작 가이드
- 실제 사용 흐름을 정리한 how-to 문서
- 운영 정책, 제한 사항, FAQ 문서

## 한 문장 요약

Harness Docs는 "왜 필요한가"와 "어떻게 쓰는가"를 함께 전달하는 고객-facing 문서 운영 제품입니다.
