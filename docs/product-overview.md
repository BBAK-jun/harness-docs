# 제품 소개

Harness Docs는 작은 cross-functional product team을 위한 제품 문서 워크스페이스입니다.

이 제품은 단순히 문서를 저장하는 곳이 아니라, 아래 흐름을 한곳에서 연결하는 것을 목표로 합니다.

- 제품 요구사항 정리
- 사용자 흐름 설명
- 기술 설계 문서화
- 정책과 의사결정 기록
- 검토와 배포 흐름 연결

## 어떤 팀을 위한 제품인가

Harness Docs는 다음과 같은 팀을 위한 제품입니다.

- PM, 디자인, 엔지니어링이 서로 다른 문서를 함께 다루는 작은 제품 팀
- 요구사항, UX 흐름, 기술 설계, 정책 문서를 따로 두되 서로 어긋나지 않게 관리하고 싶은 팀
- GitHub를 배포 채널로 쓰되, 문서의 최신 상태와 검토 맥락은 앱에서 관리하고 싶은 팀

## 어떤 문제를 풀려고 하는가

많은 팀은 제품 문서를 운영할 때 아래 문제를 반복해서 겪습니다.

- 요구사항과 UX 문서, 기술 문서가 서로 따로 관리되어 내용이 어긋남
- 문서가 최신인지 직접 비교해봐야 해서 검토 비용이 큼
- 실제로 배포된 문서와 팀 안의 검토 상태가 분리되어 운영 부담이 커짐
- 제품을 잘 아는 몇 사람에게 문서 관리가 과하게 몰림

Harness Docs는 이 문제를 문서를 "운영 대상"으로 다루는 방식으로 풀려고 합니다.

## 제품이 제공하는 핵심 가치

### 1. 문제 공감에서 사용 가이드까지 한 흐름으로 연결

한 제품을 설명하는 문서는 한 종류가 아닙니다. 요구사항, 사용자 흐름, 기술 설계, 정책 문서가 함께 맞물려야 팀이 같은 방향으로 움직일 수 있습니다.

Harness Docs는 이 문서들을 같은 워크스페이스 안에서 연결해 관리합니다.

### 2. 제품 문서 운영에 맞는 워크스페이스

각 워크스페이스는 하나의 GitHub docs 저장소와 연결됩니다. 팀은 이 안에서 문서를 작성하고 검토한 뒤, 배포 가능한 상태로 정리합니다.

즉 GitHub는 최종 배포 채널이고, 문서의 작업 맥락과 검토 이력은 앱이 관리합니다.

### 3. 팀이 같이 관리할 수 있는 문서 품질 흐름

문서는 한 번 쓰고 끝나는 자산이 아니라 계속 손봐야 하는 운영 대상입니다.

Harness Docs는 아래 같은 흐름을 제품 안에서 다룹니다.

- 어떤 문서가 최근 변경의 영향을 받았는지
- 어떤 문서가 다시 확인이 필요한지
- 어떤 문서가 최신 상태인지
- 어떤 문서가 배포 준비가 되었는지

### 4. AI를 초안 도구로 사용

AI는 최종 결정을 대신하지 않습니다. 대신 아래 작업을 빠르게 시작하게 돕습니다.

- 문서 초안 작성
- 문서 구조 제안
- 빠진 내용 점검
- 연결된 문서의 보강 포인트 정리

## 문서 예시

Harness Docs 안에서는 아래 같은 문서를 운영할 수 있습니다.

- 제품 요구사항 문서
- 사용자 흐름 문서
- 기술 설계 문서
- 정책과 의사결정 문서

## 한 문장 요약

Harness Docs는 역할별 제품 문서를 연결하고, 어떤 문서가 최신인지 더 쉽게 알 수 있게 해주는 제품 문서 운영 시스템입니다.
