# 누구를 위한 제품인가

Harness Docs는 고객-facing 문서를 제품처럼 운영해야 하는 팀을 위한 도구입니다.

특히 아래 팀에 잘 맞습니다.

- 제품을 소개하는 문서와 실제 사용 가이드를 함께 운영하는 팀
- 고객 온보딩, 설정 가이드, 운영 문서, FAQ를 흩어지지 않게 관리하고 싶은 팀
- GitHub로 문서를 배포하지만, 문서 상태와 검토 이력은 별도로 관리하고 싶은 팀

## 이런 상황에 적합합니다

### 제품 설명과 사용 가이드가 분리돼 있는 팀

많은 팀은 "왜 이 제품이 필요한가"를 설명하는 문서와 "어떻게 쓰는가"를 설명하는 문서를 따로 운영합니다.

이 경우 고객은 제품 가치를 이해하기 전에 설정 가이드만 보거나, 반대로 제품 소개만 읽고 실제 시작 방법은 놓치기 쉽습니다.

Harness Docs는 이 두 종류의 문서를 같은 문서 허브 안에서 연결해 관리하려는 팀에 적합합니다.

### 고객이 반복해서 같은 질문을 하는 팀

온보딩, 운영, 문제 해결 문서가 일관되지 않으면 고객은 같은 질문을 여러 번 하게 됩니다.

Harness Docs는 getting started, how-to, FAQ, 운영 정책 문서를 같은 흐름으로 관리하려는 팀에 적합합니다.

### 문서를 배포하지만 운영 흐름은 따로 필요한 팀

GitHub는 좋은 배포 채널이지만, 문서 검토 상태나 업데이트 필요 여부까지 자연스럽게 관리해주지는 않습니다.

Harness Docs는 GitHub를 최종 배포 채널로 쓰면서도, 문서 작업 흐름은 제품 안에서 관리하려는 팀에 적합합니다.

## 이런 역할이 함께 쓰기 좋습니다

- PM: 고객이 어떤 문제를 겪는지 설명하는 문서 작성
- 솔루션/CS 팀: 고객이 실제로 막히는 지점과 FAQ 보강
- 운영팀: 정책, 제한 사항, 운영 기준 정리
- 제품 마케팅/교육팀: 제품 가치 전달과 시작 가이드 정리

## 이런 팀에는 덜 맞을 수 있습니다

- 고객-facing 문서보다 내부 개발 문서가 주된 목적일 때
- 배포 없는 개인 메모 저장소가 필요할 때
- 문서 검토 흐름보다 단순 CMS 업로드가 더 중요한 경우

## 한 문장 요약

Harness Docs는 고객이 제품을 이해하고 실제로 써보는 데 필요한 문서를 한 흐름으로 운영하려는 팀을 위한 제품입니다.
