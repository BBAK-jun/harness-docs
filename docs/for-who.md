# 누구를 위한 제품인가

Harness Docs는 역할별 제품 문서를 함께 운영해야 하는 작은 제품 팀을 위한 도구입니다.

특히 아래 팀에 잘 맞습니다.

- PM, 디자인, 엔지니어링이 서로 다른 문서를 함께 다루는 팀
- 요구사항, 사용자 흐름, 기술 설계, 정책 문서를 흩어지지 않게 관리하고 싶은 팀
- GitHub로 문서를 배포하지만, 문서의 최신 상태와 검토 이력은 별도로 관리하고 싶은 팀

## 이런 상황에 적합합니다

### 역할별 문서가 자주 어긋나는 팀

많은 팀은 요구사항, UX 문서, 기술 문서, 정책 문서를 따로 관리합니다.

이 경우 같은 제품을 설명하면서도 문서마다 기준이 달라지고, 어느 문서가 최신인지 다시 확인하는 비용이 커집니다.

Harness Docs는 이런 문서를 같은 문서 허브 안에서 연결해 관리하려는 팀에 적합합니다.

### 최신 상태를 직접 확인하는 비용이 큰 팀

제품 변경이 잦을수록 문서도 자주 영향을 받습니다. 그런데 어떤 문서를 먼저 다시 봐야 하는지 알기 어려우면 검토 비용이 빠르게 커집니다.

Harness Docs는 문서를 하나씩 직접 대조하지 않아도 다시 확인이 필요한 문서를 빠르게 파악하고 싶은 팀에 적합합니다.

### 배포와 검토 흐름을 함께 보고 싶은 팀

GitHub는 좋은 배포 채널이지만, 팀 안에서 어떤 문서가 충분히 검토되었는지나 어떤 문서가 아직 확인이 더 필요한지까지 자연스럽게 보여주지는 않습니다.

Harness Docs는 GitHub를 최종 배포 채널로 쓰면서도, 문서 작업 흐름은 제품 안에서 관리하려는 팀에 적합합니다.

## 이런 역할이 함께 쓰기 좋습니다

- PM: 요구사항과 문제 정의 정리
- 디자이너: 사용자 흐름과 상호작용 정리
- 개발자: 기술 설계와 제약 정리
- 리드와 의사결정자: 정책과 기준 정리

## 이런 팀에는 덜 맞을 수 있습니다

- 개인 메모보다 팀 문서 운영이 중요한 경우가 아닐 때
- 배포 없는 개인 메모 저장소가 필요할 때
- 문서 검토 흐름보다 단순 CMS 업로드가 더 중요한 경우

## 한 문장 요약

Harness Docs는 역할별 제품 문서를 한 흐름으로 운영하고, 문서의 최신 상태를 더 쉽게 파악하려는 팀을 위한 제품입니다.
